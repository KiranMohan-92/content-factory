Content Factory v2

Agentic content that
survives hostile review.

A 4-phase production system built on David Deutsch's epistemological framework. Not a prompt chain — a systematic error-elimination pipeline where each agent can only criticise completed work from the phase before it.

The Problem

Single-shot AI content
has no error correction.

Generic AI generates plausible-sounding prose without mechanism. It can't distinguish a good explanation from a confident-sounding bad one — because there's no criticism phase, no hostile review, no quality gate.

🎭
No error detection
Single-shot generation has no mechanism to catch unfalsifiable claims, selection bias, or missing counterarguments.
🪞
Confident mediocrity
LLMs produce high-confidence prose regardless of explanatory quality. There's no quality gate, so everything passes.
📉
No criticism possible
Deutsch: good explanations survive hostile criticism. AI content never faces it — so it never has to be good.
🔁
The fix: sequential agents
Each phase produces completed work. The next phase criticises it. Error correction happens at every stage.
Phase 1

Research

The Researcher doesn't just search — it first classifies the topic along four dimensions to generate a precise prompt, then runs the Gemini CLI in live search mode against academic, contrarian, and industry sources simultaneously.

1
Dynamic Prompt Generator
Classifies domain (technical/philosophical/practical), depth (surface/deep), controversy (settled/polarised), and temporal sensitivity — then crafts a targeted research prompt accordingly.
2
Gemini CLI — Live Web Search
Executes the prompt via Gemini CLI in headless mode, pulling from academic papers, industry reports, primary sources, and multi-modal content (PDFs, datasets, video transcripts).
3
Research Validator
Three gates must all pass: source diversity (3+ domains), steelman requirement (minority views represented), and evidence hygiene (all claims traceable to primary sources). Fails → re-queries Gemini.
"The growth of knowledge depends entirely on the existence of disagreement."
— David Deutsch · The Beginning of Infinity
Phase 2

Analysis

The Analyzer applies Deutsch's epistemological framework to the complete research output. It doesn't summarise — it stress-tests every explanation against five distinct failure modes that separate good explanations from plausible-sounding ones.

TestWhat it catchesFailure action
Hard-to-VaryExplanations where details can change arbitrarily without breaking the theoryRebuild with fixed mechanism
FalsifiabilityClaims with no conceivable refutation — unfalsifiable by designAdd scope conditions or drop claim
Best ExplanationAccepting first plausible explanation when a simpler rival existsDocument rival, compare evidence
ArbitrarinessDecorative claims that don't affect the explanation's logical structureRemove or make load-bearing
Problem-Solution FitExplanations that answer a different question than the one posedRealign explanation to stated problem

All 5 tests must pass before the analysis.md is handed to the Writer. The Analyzer loops internally — it doesn't escalate failures, it fixes them.

Phase 3

Writing

The Writer translates validated analysis into six output formats simultaneously. Each format has a dedicated template with format-specific rules — an article is not a thread is not a LinkedIn post. The same insight, six different rhetorical structures.

FormatTemplateConstraint
📄 Long-form Articlearticle-template.mdHook → insight chain → call to think
🐦 Twitter/X Threadtwitter-thread-template-v2.mdTweet 1 = full argument, 2–22 = proof
💼 LinkedIn Postlinkedin-post-template.mdContrarian opener, no jargon, CTA
📧 Newsletternewsletter-template.mdReader-first, one actionable takeaway
🎬 YouTube Scriptyoutube-script-template.mdVisual hooks, retention loops every 90s
🎙️ Podcast Notespodcast-notes-template.mdTimestamps, quotes, chapter markers

If a draft fails the Writer's own internal validator, it revises before passing to the Editor. The Editor never sees a draft the Writer itself wouldn't approve.

Phase 4

Editing

The Editor runs a multi-pass review across all six formats: fact-verification, logic integrity, style consistency, engagement quality, and format-specific adherence. Content must score ≥ 9.3/10 — or it returns to the Writer with ranked, specific fixes.

🔬
Fact Verification
Every factual claim is traced back to a research.md source. Unsourced claims are flagged critical and must be removed or cited.
🔗
Logic Integrity
The argument chain is tested for validity. Gaps between premises and conclusions are returned as blocking issues.
🎯
Engagement Quality
First 100 words, hook strength, pacing, and ending. Reader must not be able to stop reading after the first sentence.
📐
Format Adherence
Each format is validated against its template's structural rules. A LinkedIn post that reads like an article fails immediately.

Threshold: 9.3 / 10. If met, drafts advance to Phase 4.5 — the CODEX Brutal Critic. The Editor's approval is necessary but not sufficient for publication.

Phase 4.5

CODEX Brutal Critic

The final gatekeeper before publication. Powered by OpenAI's Codex CLI in headless mode. Its job: find what the Editor missed. The threshold is 9.5 — higher than the Editor's 9.3 — because the gap between excellent and world-class is exactly where this agent lives.

Claim Stress-Test
What would a hostile domain expert immediately attack?
Counterexample Search
Generalisations that break under real-world cases.
Evidence Sufficiency
Strong claims require proportionate evidence strength.
Hard-to-Vary Test
Would the argument survive any detail change?
Falsifiability Check
What evidence, in principle, could prove it wrong?
Originality Test
Genuinely new or rearranged familiar insights?
Intellectual Charity
Are the best versions of opposing views engaged?
Predictive Power
Does the explanation reach into adjacent domains?
1
Iteration 1
Full CODEX CLI review. Ranked issues returned to Writer if score < 9.5.
2
Iteration 2
Re-run CODEX on revised content. Focus on remaining blocking issues.
3
Escalation
After 3 failures: human decision required. No automatic approval below 9.5.
Orchestrator

State Machine &
Recovery

The orchestrator runs as a state machine with full checkpointing at every phase boundary. If anything fails — API error, validation failure, network timeout — the pipeline resumes from the last saved checkpoint, not from scratch.
